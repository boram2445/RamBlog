# 로드맵 구간(주차/Day) task 문서 작성

roadmap의 한 구간 상세 task 문서를 생성·개편한다. **프로젝트마다 계획 레이아웃이 다르므로, 기존 관례를 먼저 감지해 따른다.**

## STEP 0 — 프로젝트 계획 레이아웃 감지 (하드코딩 금지)

1. **roadmap 디렉터리·파일 네이밍 감지**: `docs/**/roadmap*/`·`docs/plan/**`에서 기존 `week-N.md` 또는 `weekN.md`를 찾아 **그 경로·네이밍을 그대로 따른다**. 없으면 사용자에게 확인.
2. **단위 자동 판단**: 기존 파일/ROADMAP을 보고 결정 —
   - **주차형**: Week = Day 1~7, 매주 리셋.
   - **연속 Day형**: Day 번호가 1~N 계속 이어짐(서브데이 `41b` 등 허용).
3. **톤 참고**: 가장 최근 구간 문서를 정본 예시로 톤·형식을 맞춘다.

## 산출물 스켈레톤

```
# {구간명} — {주제}

## 이번 구간 목표 (또는 상단 요약)
{끝나면 무엇이 달라지는지. 정량 기준 포함}

## 요약 표
| Day | 목표 | 주요 산출물 | 완료 |     ← 주차형 (날짜 열 없음)
                                          연속 Day형은 프로젝트 관례 따름
## Day N — {제목}
**목표** — {한 줄}
| 그룹 | 할 일 | 구현 방식 | 상태 |
> 산출물 / ⚠️ 노트

## 백엔드 의존 — 대기 항목 (백엔드 연동 구간에만)
### ⓐ 아예 없는 엔드포인트   | 엔드포인트 | 필요 화면(URL) | 대기 중인 기능 | 예정 |
### ⓑ 필드가 부족           | 엔드포인트 | 필요 화면(URL) | 필요한 변경 | 대기 중인 기능 |

## 완료 기준
```

## task 표 규칙 — `그룹 | 할 일 | 구현 방식 | 상태`

- **할 일** = 자연어 서술형(무엇을·왜). 사람이 읽는 칸. 함수명·타입명·파일경로 등 코드 식별자 금지.
- **구현 방식** = 어떻게 붙일지 한 구절. 남긴다: 핵심 엔드포인트(`GET /api/...`)·접근법(서버/클라 컴포넌트·fetch 방식·낙관적 토글·미들웨어·병렬)·명령어·env. 뺀다: 함수명·타입명·파일경로·(신규/삭제) 마커 — 구현 시점에 정한다.
- 수동 확인/QA 행의 구현 방식은 `—`.

예시:
```
| 리뷰 목록 | 그 가게 리뷰를 페이지 단위로 따로 불러온다 | 상세와 별도 엔드포인트라 병렬 호출 · GET /api/ratings/restaurant/{id} | [ ] |
```

## 입도 규칙 — 잘게 쪼개지 않기

- 기능 단위 1행. 타입 정의 + API 함수는 통합.
- 같은 화면의 부수작업(mock 삭제 등)은 주작업 행에 흡수.
- 반복형(에러 페이지 N종, 인증 가드 N단계)은 한 행으로, 세부는 구현 방식에.
- 단, 레이어(API 함수 / 상태관리 훅 / UI 배선)는 행 분리 유지.
- 완료된 이력은 기록이므로 더 공격적으로 병합.

## 진행 표기

- task는 `[x]`/`[ ]` (상태 열).
- 완료된 Day는 **상단 요약 행 앞에 ✅ 이모지** 추가.

## 백엔드 계약 정합 — 준비 상태부터 판별

작성 전 백엔드가 어디까지 됐는지 확인한다. **아직 없을 수 있다.**

| 백엔드 상태 | 구간 성격 | task 작성 |
|---|---|---|
| **미설계** (API 없음) | 프로토타입/mock | 필요 API 계약을 정의 → "백엔드에 요청할 API" 목록. 화면은 mock 우선. |
| **스펙만** (문서 O·코드 X) | mock/설계 검증 | 스펙 대조로 설계, 각 API "미구현(대기)" 표시. mock 우선·도착 시 교체. |
| **구현됨** (코드 O) | 통합 (mock→실 API) | **코드(컨트롤러·DTO) 직접 대조.** 스펙 문서만 믿지 않기. 아래 절차. |

**구현됨 절차**: ① 없는 엔드포인트 전제 금지 — 부족분은 ⓐ 엔드포인트 없음 / ⓑ 필드 부족으로 분류해 대기 표. ② 각 대기 항목에 막히는 화면 URL. ③ per-user 상태 필드(찜/팔로우 여부 등 optional-auth)가 공개 GET에 있는지 점검. ④ 프론트 자체 데이터소스(임시 어댑터·합성 mock) 제거 대상 명시, 렌더링 SDK와 데이터 API 구분. ⑤ 완료 약속(화면 수·mock 0)과 실제 커버리지 일치 검증.

**미설계/스펙만 절차**: 화면 task는 mock 기반. 필요 계약은 "요청 API" 표(엔드포인트·필요 화면 URL·이유)로 분리. mock→실 API 교체는 백엔드 도착 구간으로 미룸.

## 리스크 배치

- 외부/백엔드 의존·불확실 작업은 크리티컬 패스에서 분리 → 마지막 "도착 시" 버킷. 안 오면 다음 구간.
- 릴리즈 게이트(mock 정리·빌드)는 불확실 작업보다 **앞에**.
- 공유 함수는 먼저 쓰는 Day에 만들고 이후 재사용.
- 한 Day에 신규 기능 + 불확실 작업 + 릴리즈 게이트를 몰지 않기.
- 클라 훅 첫 사용 구간엔 provider 셋업 선행 노트.

## 자가 검증

- [ ] 계획 레이아웃(디렉터리·파일명·단위)을 기존 관례에 맞췄는가
- [ ] 백엔드 준비 상태(미설계/스펙만/구현됨)를 판별했는가
- [ ] (구현됨) 엔드포인트가 코드에 실재하는가 / (미설계·스펙만) 필요 API를 mock 우선 + 요청·대기로 표시했는가
- [ ] 할 일 칸에 코드 식별자가 없는가 · 구현 방식 칸은 접근법+엔드포인트만인가
- [ ] Day마다 `**목표**` 한 줄, 완료 Day엔 ✅
- [ ] 대기 표 각 항목에 필요 화면(URL)이 있는가
- [ ] 릴리즈 게이트가 "도착 시" 버킷보다 앞인가
