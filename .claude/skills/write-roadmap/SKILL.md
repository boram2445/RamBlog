# 전체 로드맵 (`ROADMAP.md`) 작성

여러 구간(주차 또는 연속 Day)을 아우르는 로드맵을 생성한다. 구간 상세(`week*.md`)는 이후 **write-week-task 스킬**로 만든다. **프로젝트마다 성격·레이아웃이 다르므로 먼저 감지·판별한다.**

## STEP 0 — 입력 유형 자동 판별 (로드맵의 근거)

로드맵은 근거 문서 없이 쓰지 않는다. 프로젝트가 어느 쪽인지 먼저 판별:

| 유형 | 근거 입력 | 없을 때 |
|---|---|---|
| **신규 구축** (제품을 새로 만듦) | **PRD**(`docs/PRD.md`·`**/*기획*` 등) | PRD를 요청 (① 경로 ② 붙여넣기 ③ 링크). "임시로"면 핵심 가치·화면·완료 기준·기간을 질문으로 수집 |
| **기존 개선/리팩터** (레거시 정비) | **진단·부채 분석** (현 문제 영역 목록. 기존 `ROADMAP.md` Context의 진단 표 등) | 진단을 먼저 함께 만든다 — 영역별(보안/성능/SEO/DX 등) 핵심 문제를 표로 |

- 프로젝트 성격이 애매하면 사용자에게 "신규 구축인가요, 기존 개선인가요?"부터 확인.
- 계획 디렉터리·파일 네이밍(`docs/plan/roadmap1` vs `docs/roadmap1`, `week-N` vs `weekN`)과 **단위(주차형 vs 연속 Day형)**는 기존 파일을 감지해 따른다. 없으면 확인.

## STEP 1 — 근거에서 추출

- **신규**: 핵심 가치(우선순위) · 화면 목록(라우트) · 접근 권한(공개/인증/보호) · MVP 범위(포함/제외) · 완료 기준 후보.
- **개선**: 부채 영역(카테고리별 핵심 문제) · 개선 트랙 · 리스크·제약(배포 차단 등) · 완료 기준 후보.

## STEP 2 — 현재 상태 조사

- 기존 코드가 있으면 화면/영역별 현재 상태(완료/진행/미착수).
- **백엔드 준비 상태 판별** (write-week-task와 동일): 미설계 / 스펙만 / 구현됨 — 통합 구간 배치 근거.

## STEP 3 — ROADMAP.md 스켈레톤

```
# {프로젝트} — {로드맵 제목}
{목표 한 줄}

## Context / 현재 상황     ← 개선형이면 진단 표
## 완료 기준               ← 정량·체크박스
## (화면 or 영역) 현재 상태
## 라우트 맵 / 영향 범위
## 표준 절차               ← 반복 절차가 있으면
## 구간별 요약             ← 구간(주차/Day 묶음)당 한 줄: 목표 + 산출물
## 검증 게이트             ← 구간 경계 통과 조건
```

- **단위**는 STEP 0에서 감지한 대로(주차형이면 Week 요약, 연속 Day형이면 Day 일정).
- **시퀀싱**: 시각 프로토타입 먼저 → 데이터 레이어 → 통합 → QA·배포. 불확실/외부(백엔드) 의존은 뒤 구간. 백엔드가 준비돼 있으면 통합 구간을 앞당김.

## STEP 4 — 구간 상세로 연결

ROADMAP.md 확정 후, 각 구간 문서(`week*.md`)는 **write-week-task 스킬**로 생성. 표 형식·입도·백엔드 계약 정합·리스크 배치는 그 스킬 담당 — 여기서 중복 기술 금지.

## 자가 검증

- [ ] 입력 유형(신규=PRD / 개선=진단)을 판별하고 근거를 확보했는가
- [ ] 완료 기준이 정량인가
- [ ] 모든 화면/부채 영역이 어느 구간엔가 배정됐는가 (누락 없음)
- [ ] 백엔드 준비 상태가 구간 배치에 반영됐는가
- [ ] 구간 경계마다 검증 게이트가 있는가
- [ ] 계획 디렉터리·파일 네이밍·단위를 기존 관례에 맞췄는가
