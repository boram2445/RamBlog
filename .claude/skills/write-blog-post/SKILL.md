## 입력 파싱

인자를 보고 모드를 결정한다:

- `day <N>` — 30일 회고 시리즈 모드. Day N의 블로그 편 기준
- `post <N>` — 30일 회고 시리즈 모드. N편 기준 (day 별칭)
- `--short` — 길이 옵션: 800~1500자
- `--long` — 길이 옵션: 3000자 이상
- 길이 옵션 없으면 기본 1500~2500자
- 그 외 자유 텍스트 — 범용 모드

## 시리즈 모드 실행 흐름

### 1. 컨텍스트 수집

다음을 병렬로 읽는다:

**로드맵 매핑**
`docs/roadmap1/ROADMAP.md`의 "블로그 시리즈 주제" 표를 읽어서:

- 인자가 `day <N>`이면 해당 Day가 속한 주차 파일(`docs/roadmap1/week{N}.md`)을 읽어 "블로그 X편 발행" 여부와 편 번호를 확인한다
- 인자가 `post <N>`이면 시리즈 N편이 어느 Day들에 걸쳐 있는지 역방향으로 매핑한다
- 해당 편이 다루는 작업 Day 범위를 확정한다 (예: 2·3편은 Day 8~11을 회고)

**관련 주차 파일**
위에서 확정한 Day 범위가 속한 `docs/roadmap1/week{N}.md`를 읽어 해당 Day들의 작업 상세를 파악한다. 없으면 없는 대로 진행.

**git log**
해당 Day들의 실제 구현이 커밋되어 있다면:

```
git log --oneline --since="..." -- src/
```

또는 `git log --oneline <범위>` 로 커밋 목록과 diff 요약을 수집한다. git log가 비어있거나 작업 미완료면 plan 정보만으로 진행하고 나중에 코드 예시를 채워야 함을 글 최하단에 TODO로 남긴다.

### 2. 글 구조 조립

아래 템플릿을 사용한다. 헤더는 텍스트만, 이모지 사용 금지.

```markdown
---
title: <제목>
series: RamBlog 30일 회고
part: <편 번호>
draft: true
---

<1인칭 도입. 작업 동기와 배경을 한 문단으로. "3년 전 내", "내 코드는 왜" 같은 1인칭 회고체. 단정형 종결("~다").>

## 문제 진단

<정량 지표(영향 받는 파일 수, 호출 횟수 등)와 왜 이게 문제인지(보안/성능/유지보수 관점 1~2가지)>

## 해결 과정

### 기존 코드

\`\`\`ts
// 기존
// ...
\`\`\`

### 변경 후

\`\`\`ts
// 변경 후
// ...
\`\`\`

<필요 시 단계별 설명 추가. 코드 비교 최소 1쌍, 권장 2~4쌍.>

## 배운 점

<신입 시절의 나는 왜 그렇게 작성했나. 지금이라면 어떻게 다르게 할까. 단정적 진단("이건 IDOR이다")과 겸손한 회고("그땐 인증과 인가가 같은 줄 알았다")의 균형.>

<시리즈 모드: 다음 편 한 줄 예고. 범용 모드: 생략.>
```

### 3. 파일 저장

`.claude/posts/<두자리 편번호>-<영문-slug>.md` 형식으로 저장.

- 시리즈 모드: `.claude/posts/01-next-13-to-16-migration.md`
- 범용 모드: `.claude/posts/<날짜>-<slug>.md` (오늘 날짜 YYYYMMDD)

파일 저장 후 대화창에 다음을 출력한다:

- 저장된 파일 경로
- 글 길이 (자)
- 참조한 컨텍스트 출처 목록
- 사용자가 직접 보완해야 할 부분 1~2개 (TODO가 있으면 명시, 없으면 코드 예시 검토 권고)

## 범용 모드 실행 흐름

시리즈 특유 표현("신입 시절의 나", "다음 편 예고")은 생략.
나머지 구조(도입 → 문제 진단 → 해결 → 배운 점)와 톤 규칙은 동일하게 적용.
사용자가 prompt에 코드 스니펫이나 링크를 붙였으면 그것을 Before/After 예시의 기반으로 활용한다.

frontmatter에서 `series`와 `part` 필드는 생략한다.

## 톤 규칙 (항상 적용)

- 1인칭 회고체: "3년 전 내", "내 코드는 왜", "지금의 나라면"
- 반-격식: 단정형 종결("~다"), 구어체 단어 허용("죄악", "함정")
- 이모지 금지: 헤더, 본문, 코드 주석 어디에도 이모지를 쓰지 않는다
- 코드 주석은 `// 기존`, `// 변경 후` 같은 텍스트 사용
- 금지: "오늘은 ~에 대해 알아보겠습니다" 류 도입, Claude/AI 언급, "본 포스트에서는" 같은 격식체
