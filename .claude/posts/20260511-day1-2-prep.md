3년 묵은 RamBlog를 다시 열고 30일 개선 플랜을 짰는데, 정작 첫 두 날은 코드 리팩터가 아니라, 안전장치 두 개를 까는 일이었다. main에 커밋하면 Vercel이 자동 배포해 버리는 환경에서 미완성 변경분을 production에 흘려보내지 않을 방법, 그리고 환경변수가 빠졌을 때 사용자가 먼저 발견하는 구조를 뒤집는 것.

## 문제 진단

### 배포 격리가 없었다

main 브랜치에 push하면 Vercel이 자동으로 production에 배포된다. 30일 동안 작업하면서 중간 상태의 코드가 그대로 배포되면 안 됐다. 일단 별도 브랜치가 필요했다.

### 환경변수 7곳이 흩어져 있었다

`process.env.X` 가 4개 파일 7곳에 있었다.

| 파일                                        | 참조 수 |
| ------------------------------------------- | ------- |
| `src/service/sanity.ts`                     | 4       |
| `src/service/image.ts`                      | 1       |
| `src/app/api/auth/[...nextauth]/options.ts` | 2       |
| `src/app/layout.tsx`                        | 1       |

사실 더 큰 문제가 있었다. `process.env.X` 의 타입은 `string | undefined` 인데, 코드 어디에서도 그 `undefined` 를 제대로 처리하지 않고 있었다. 일부는 `|| ''` 폴백을 붙여 빈 문자열을 SDK에 넘겼고, 일부는 `undefined` 를 그냥 통과시켰다.

`undefined` 가 들어간 Sanity 클라이언트는 일단 뜨긴 하는데 첫 fetch에서 정체불명의 에러를 뱉는다. 빈 문자열이 들어간 Google OAuth는 로그인을 눌러봐야 실패한다. env가 빠진 걸 빌드 때 못 잡으면, 결국 사용자가 먼저 이상한 걸 발견하게 된다.

## 해결 과정

### Day 1 — 작업 브랜치 + .env.example

`git checkout -b refactor/migration` 으로 30일 작업 브랜치를 만들었다. main에 머지하지 않는 한 Vercel auto-deploy는 트리거되지 않는다. 기존 production은 3년 치 부채를 그대로 안고 살아 있지만, 새로운 부채가 추가되지는 않는다. 30일 동안 안심하고 빌드를 깨먹을 수 있는 자유가 거기서 생겼다.

`.env.example` 도 함께 작성했다. Sanity 3개, NextAuth 2개, Google OAuth 2개, SEO 1개 — 총 8개 변수. 각 변수 위에 출처나 생성 방법을 한 줄 주석으로 달았다.

```
# --- Sanity ---
# Sanity 프로젝트 ID (https://www.sanity.io/manage)
SANITY_PROJECT_ID=
# 자산 업로드/뮤테이션용 write 토큰
SANITY_SECRET_TOKEN=

# --- NextAuth ---
# JWT 서명용 시크릿 — `openssl rand -base64 32` 로 생성
NEXTAUTH_SECRET=
```

### Day 2 — zod 단일 검증 모듈

환경변수 검증에 zod를 선택한 이유는 세 가지다.

첫째, **타입 추론**. `z.infer<typeof envSchema>` 가 검증 통과 후의 타입을 자동 생성한다. 검증을 통과한 시점부터 `env.SANITY_PROJECT_ID` 는 `string` 이다. `|| ''` 폴백이 필요 없어진다.

둘째, **`.env.example` 과 1:1 대응**. required 변수는 `z.string().min(1)`, 선택 변수는 `.optional()`, `NEXTAUTH_URL` 같은 URL은 `.url()`. 컨트랙트와 검증이 같은 언어로 쓰인다. `.env.example` 을 수정하면 스키마도 수정해야 한다는 게 코드로 표현된다.

셋째, **검증 라이브러리 통일**. 30일 플랜 후반부에 API body 검증과 보안 패치에 어차피 zod를 도입할 예정이다. env만 별도 라이브러리(envalid 등)로 두면 프로젝트 안에 검증 도구가 두 개가 된다.

대안도 검토했다. 호출부마다 `if (!process.env.X) throw` 를 박는 방법은 반복적이고 포맷 검증이 불가능하다. TypeScript의 `declare namespace NodeJS { interface ProcessEnv {...} }` 는 컴파일 타임 타입 힌트만 줄 뿐, 런타임에 실제 값이 들어있는지는 전혀 검증하지 않는다.

Next.js 전용 env 라이브러리인 `@t3-oss/env-nextjs` 도 잠깐 봤다. 이 라이브러리의 핵심 역할은 `NEXT_PUBLIC_*` 변수(클라이언트 번들에 인라인되는 공개 변수)와 서버 전용 변수를 분리해서 검증하는 것인데, RamBlog에는 `NEXT_PUBLIC_*` 변수가 없기 때문에 `import 'server-only'` + zod로 충분하다. 이 라이브러리를 쓰면 오히려 `runtimeEnv` 객체를 별도로 선언해야 해서 코드가 더 늘어난다. 클라이언트에서도 읽어야 하는 변수가 생기면 그때 도입을 고려하기로 했다.

`src/lib/env.ts` 의 전체 내용:

```ts
import "server-only";
import { z } from "zod";

const envSchema = z.object({
  SANITY_PROJECT_ID: z.string().min(1),
  SANITY_DATASET: z.string().min(1),
  SANITY_SECRET_TOKEN: z.string().min(1),
  NEXTAUTH_URL: z.string().url(),
  NEXTAUTH_SECRET: z.string().min(1),
  GOOGLE_OAUTH_ID: z.string().min(1),
  GOOGLE_OAUTH_SECRET: z.string().min(1),
  GOOGLE_VERIFICATION_ID: z.string().optional(),
});

const result = envSchema.safeParse(process.env);
if (!result.success) {
  const formatted = result.error.issues
    .map((issue) => `  - ${issue.path.join(".")}: ${issue.message}`)
    .join("\n");
  throw new Error(`Invalid environment variables:\n${formatted}`);
}

export const env = result.data;
export type Env = z.infer<typeof envSchema>;
```

모듈 톱레벨에서 검증하니까 `env` 를 처음 import 하는 순간 바로 터진다. 변수 하나가 빠지면 이렇게 뜬다.

```
Error: Invalid environment variables:
  - SANITY_PROJECT_ID: String must contain at least 1 character(s)
```

어느 SDK가 몇 번째 요청에서 조용히 깨지는지 뒤쫓을 필요가 없다. `import 'server-only'` 를 앞에 붙여 클라이언트 번들에 섞이는 것도 막았다.

호출부는 이렇게 바뀌었다:

```ts
// 기존 — src/service/sanity.ts
projectId: process.env.SANITY_PROJECT_ID,  // undefined 가능
dataset: process.env.SANITY_DATASET,
token: process.env.SANITY_SECRET_TOKEN,

// 변경 후
import { env } from '@/lib/env';
projectId: env.SANITY_PROJECT_ID,  // string — 빌드 통과 = 검증 완료
dataset: env.SANITY_DATASET,
token: env.SANITY_SECRET_TOKEN,
```

```ts
// 기존 — src/app/api/auth/[...nextauth]/options.ts
clientId: process.env.GOOGLE_OAUTH_ID || '',     // 빈 문자열을 OAuth ID로 넘김(!)
clientSecret: process.env.GOOGLE_OAUTH_SECRET || '',

// 변경 후
clientId: env.GOOGLE_OAUTH_ID,
clientSecret: env.GOOGLE_OAUTH_SECRET,
```

작업 후 `grep -rn "process\.env\." src/` 를 돌렸을 때 0건이 나왔다. `env.ts` 안에서 `process.env` 를 bare object로 zod에 넘기는 한 줄 외엔 직접 참조가 완전히 사라진 상태다.

## 배운 점

신입 시절의 나는 환경변수를 "필요한 시점에 `process.env` 에서 꺼내 쓰면 그만" 이라고 생각했다. `|| ''` 로 폴백을 붙이면 TypeScript 에러가 사라지니까 맞는 코드라고 생각했다. 그런데 빈 문자열을 받은 SDK가 어떻게 반응하는지는 그 SDK를 열어봐야 알았다. 어떤 SDK는 조용히 통과시키고, 어떤 SDK는 런타임 첫 요청에서 터진다.

`.env.example` → `env.ts` 스키마 → 호출부 `env.X` 세 곳이 같은 컨트랙트를 바라보는 구조가 되면, 변수를 추가하거나 삭제할 때 세 곳을 같이 바꾸지 않으면 빌드가 막힌다. 세 파일을 같이 바꿔야 한다는 걸 내 기억에 맡기는 게 아니라, 빌드가 대신 막아주는 구조다.

작업 브랜치 분리는 "코드를 격리하는 것" 이 아니라 "배포를 격리하는 것" 이었다. main을 건드리지 않는 한 production의 3년 치 부채는 그대로 살아 있지만, 새 부채가 노출되지는 않는다. 30일 동안 매일 빌드를 안심하고 깨먹을 수 있는 자유는 거기서 나왔다.
